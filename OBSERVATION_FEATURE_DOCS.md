## 1. Feature Overview

The Observation feature allows coaches to:

1. **Schedule** a classroom observation with teacher, school, subject, grade and date details
2. **Score** the observation using either the HOTS or FICO rubric depending on region
3. **Record** the classroom session and submit audio to the Digital Coach (DC) external AI service
4. **Receive** automatic rubric scores and debrief notes generated by DC from the audio
5. **Review and edit** the auto-generated scores before submitting the observation

### Framework Selection by Region

| Region | Framework |
|---|---|
| ICT / Islamabad | FICO |
| Punjab / Rawalpindi / other | HOTS |

Framework is auto-selected from the coach's profile region when scheduling. Both frameworks are visible in the app; coaches cannot change the framework after scheduling.

## 3. Observation Frameworks

### HOTS Framework

Used in Punjab, Rawalpindi and all non-ICT regions.

**Scoring:** 6 dimensions, each scored 0–10 (Assessment & Feedback: 0–20). Total: 0–80.

| Score Range | Proficiency |
|---|---|
| 0 – 40 | Below Basic |
| 41 – 60 | Basic |
| 61 – 75 | Proficient |
| 76 – 80 | Advanced |

**Submit gate:** `total_score > 0` required.

---

### FICO Framework

Used in ICT / Islamabad region. Divided into 4 sections:

#### Section B — Structural & Pedagogical (always shown, always scored)

| Code | Indicator |
|---|---|
| SI4 | Effective Resource Use |
| SI5 | Productive Learning Time |
| SI6 | Classroom Management |
| PIC6 | Positive & Supportive Disposition |
| PIC7 | Equitable Interactions |

#### Section C — Subject Specific (auto-filtered by subject)

**Math (Concrete/Abstract stage):**

| Stage | Indicators |
|---|---|
| Concrete (C) | M3C, M4C |
| Pictorial/Abstract (PA) | M3PA, M4PA |

**Science:** S3, S4  
**Literacy/English:** L4

#### Section D — Student Engagement Evidence Markers (observed but not scored)

SE1 through SE6 — presence/absence of student engagement signals.

#### Section E — Student Sampling

5 students × 3 questions (boolean per question). Coach fills these manually.

**FICO Scoring:** Score = (earned points from B + C) / (max points from B + C) × 100%

| Score Range | Proficiency |
|---|---|
| ≥ 85% | Advanced |
| 70–84% | Proficient |
| 50–69% | Basic |
| < 50% | Below Basic |

**Submit gate:** All 5 Section B indicators must be scored (regardless of DC auto-score).

---

### Score Values

| Selection | Value |
|---|---|
| Yes | 1 |
| Partial | 0.5 |
| No | 0 |
| N/A | excluded from denominator |

## 4. Frontend Components

All observation components live in `src/components/observation/`.

### `ScheduleDialog.tsx`
Dialog for creating a new observation. Reads coach's `region` from auth profile. Auto-selects framework (`isRegionFico(region)` → FICO, else HOTS). Inserts a `Scheduled` status row into `cot_observations` with `framework` and empty `fico_rubric: {}`.

Shows a "FICO Framework" or "HOTS Framework" badge next to the region field so the coach knows which rubric will be applied.

---

### `DraftObservationsTab.tsx`
Lists all observations with `status = 'Draft'`. Each card is collapsible and contains three tabs:

| Tab | Component | Purpose |
|---|---|---|
| HOTS Rubric / FICO Rubric | `HotsRubricForm` or `FicoRubricForm` | Score the observation |
| Debrief | `DebriefForm` | Write observation notes and teacher feedback |
| DC Analysis | `DCAnalysis` | Record audio and run Digital Coach analysis |

**Local overrides:** Score changes from child forms are kept in a local `localOverrides` map so the UI reflects updates without a full page reload.

**Submit validation:**
- FICO: all Section B indicators must be scored (`isFicoSectionBComplete`)
- HOTS: `total_score > 0`

---

### `HotsRubricForm.tsx`
Slider-based form for 6 HOTS dimensions. Saves `hots_rubric` and `total_score` to the database. Calls `getProficiencyLevel` from `src/lib/observation-utils.ts` for the score banner.

---

### `FicoRubricForm.tsx`
Button-based (Yes / Partial / No / N/A) form for FICO indicators. Sections B, C (subject-filtered), D, and E are all shown.

**DC Auto-score banner:** When `observation.dc_status === 'completed'`, a blue banner appears at the top:
> "Auto-scored by Digital Coach — review and adjust if needed"

Scores are pre-populated from `observation.fico_rubric` via a `useEffect` that re-runs on `observation.id`. Coach can edit any score and save; the save overwrites the DB row.

---

### `DebriefForm.tsx`
Two textareas:
- **Observation Notes** (`hots_notes`) — DC writes section-level feedback here when analysis completes
- **Feedback for Teacher** (`notes_for_teacher`) — always manually filled by coach

**"Generated by DC" badge:** Appears next to the Observation Notes label when `dc_status === 'completed'` and `hots_notes` is non-empty.

---

### `DCAnalysis.tsx`
State machine with 7 phases:

```
idle → recording → uploading → dispatching → processing → completed
                                                        → failed
```

**On mount:** If `observation.dc_status === 'processing'`, jumps straight to the `processing` phase and subscribes to Realtime.

**Guard:** If `observation.framework !== 'FICO'`, renders an "ICT Only" message — DC is not available for HOTS observations.

**Recording:**
- `navigator.mediaDevices.getUserMedia({ audio: true })`
- Detects `audio/webm;codecs=opus` (Chrome/Firefox) vs `audio/mp4` (Safari)
- `MediaRecorder` collects chunks; on stop, builds a `Blob` and calls `uploadAudio()`

**Upload to S3:**
1. Calls `supabase.functions.invoke('dc-presign', { observation_id, mime_type })`
2. Edge function returns `presigned_put_url`, `presigned_get_url`, `s3_key`
3. Browser `PUT`s the audio blob directly to S3 (never passes through any server)

**Dispatch to DC:**
- Calls `supabase.functions.invoke('dc-start', { observation_id, s3_key, presigned_get_url, lesson_plan_text })`
- On success → moves to `processing` phase

**Processing phase:**
- Supabase Realtime subscription on `dc_analyses` WHERE `observation_id = {id}`
- Polling fallback every 8 seconds (in case Realtime misses an event)
- When status = `completed` → refreshes parent observation, transitions to `completed`
- When status = `failed` → shows error, transitions to `failed`

**Completed:** Shows collapsible sections with per-indicator score badges and DC feedback text.

## 6. Supabase Edge Functions

All functions live in `supabase/functions/`. They run on Deno runtime. Shared CORS headers are in `supabase/functions/_shared/cors.ts`.

### `dc-presign`

**Endpoint:** `POST /functions/v1/dc-presign`  
**Auth:** Requires user JWT in `Authorization` header  
**Purpose:** Generate pre-signed S3 URLs for audio upload and download

**Request body:**
```json
{ "observation_id": "uuid", "mime_type": "audio/webm" }
```

**Response:**
```json
{
  "presigned_put_url": "https://...",
  "presigned_get_url": "https://...",
  "s3_key": "dc-audio/{observation_id}/{timestamp}.webm"
}
```

**Logic:**
1. Verifies user JWT
2. Confirms observation belongs to the user
3. Reads AWS credentials from Supabase secrets (`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_REGION`, `S3_BUCKET_NAME`)
4. Uses AWS SDK v3 (`npm:@aws-sdk/client-s3` + `npm:@aws-sdk/s3-request-presigner`)
5. PUT URL expires in 30 minutes; GET URL expires in 2 hours

**S3 key format:** `dc-audio/{observation_id}/{Date.now()}.{ext}`

---

### `dc-start`

**Endpoint:** `POST /functions/v1/dc-start`  
**Auth:** Requires user JWT  
**Purpose:** Submit the uploaded audio to the DC external API for analysis

**Request body:**
```json
{
  "observation_id": "uuid",
  "s3_key": "dc-audio/...",
  "presigned_get_url": "https://...",
  "lesson_plan_text": "optional topic string"
}
```

**Response:**
```json
{ "ok": true, "dc_task_id": "abc123" }
```

**Logic:**
1. Verifies JWT and observation ownership
2. Calls DC API: `POST {DC_BASE_URL}/api/external/process-audio-from-s3` with FormData containing:
   - `audio_s3_url` — the presigned GET URL
   - `lesson_plan_text` — observation topic
   - `test_name` — `obs-{observation_id}`
   - `rubric_type` — `fico-v3`
   - `region` — `ict`
   - `teacher_id` — teacher name
   - `callback_url` — `{SUPABASE_FUNCTIONS_URL}/dc-callback`
   - `subject_shortcode` + `subject_label`
3. DC responds immediately with `{ task_id }` — actual analysis runs async
4. Inserts `dc_analyses` row with DC's `task_id`
5. Updates `cot_observations` with `dc_status = 'processing'`, `dc_task_id`, `dc_audio_s3_key`, `dc_requested_at`

**Timeout:** 30 seconds for DC API call (`AbortSignal.timeout(30_000)`)

---

### `dc-callback`

**Endpoint:** `POST /functions/v1/dc-callback`  
**Auth:** None — DC POSTs here directly; always returns HTTP 200 to stop DC retrying  
**Purpose:** Receive progress updates and final results from DC

**Request body (DC format):**
```json
{
  "task_id": "abc123",
  "status": "completed",
  "progress": 100,
  "current_step": "...",
  "message": "...",
  "error": null,
  "results": { "section_b": {...}, "section_c": {...}, "section_d": {...} }
}
```

**Logic:**
1. Finds `dc_analyses` row by `task_id`
2. Updates `dc_analyses` with latest status/progress/results
3. If `status === 'completed'`:
   - Maps DC scores → FICO rubric (`mapDcResultsToFicoRubric`)
   - Calculates score + proficiency level (`scoreFromRubric`)
   - Builds debrief notes from section feedback (`buildDebriefNotes`)
   - Updates `cot_observations` in one write: `dc_status`, `dc_results`, `dc_completed_at`, `fico_rubric`, `total_score`, `proficiency_level`, `hots_notes`
4. If `status === 'failed'`: updates `dc_status = 'failed'`, `dc_error`

## 7. Digital Coach Integration

### DC API

| Property | Value |
|---|---|
| Base URL | `http://ec2-54-179-226-216.ap-southeast-1.compute.amazonaws.com:8080` |
| Endpoint | `POST /api/external/process-audio-from-s3` |
| Auth | None (IP-based trust) |
| Pattern | Async — returns `task_id` immediately, POSTs results to `callback_url` |

### Score Mapping

DC returns scores per indicator as: `'yes' | 'partial' | 'no' | 'N/A' | 'UNABLE_TO_DETECT'`

| DC Value | FICO Rubric Value | Notes |
|---|---|---|
| `yes` | `yes` | |
| `partial` | `partial` | |
| `no` | `no` | |
| `N/A` | `na` | Excluded from score denominator |
| `UNABLE_TO_DETECT` | omitted | Shown as unscored in UI |

DC's score keys are indicator codes (SI4, SI5, PIC6, M3C, S3, L4, SE1, …) — mapped directly to FICO rubric keys.

### Debrief Auto-population

`hots_notes` is built from DC section feedback in this format:

```
[Section B — Structural & Pedagogical]
{section_b.feedback.english}

[Section C — Subject Specific]
{section_c.feedback.english}

[Section D — Student Engagement]
{section_d.feedback.english}
```

`notes_for_teacher` is **not** written by DC — coach fills this manually.

### Section E

DC does not score Section E (student sampling). The `section_e` array is written as 5 blank student objects `{q1: null, q2: null, q3: null}` — coach fills these manually.

### Race Condition Prevention

`dc-start` calls the DC API first and only inserts the `dc_analyses` row **after** receiving DC's `task_id` response. Since DC processing takes seconds-to-minutes and the DB write is milliseconds, the callback row will always exist before DC fires the callback.

## 9. End-to-End Flow

### Full coach journey (FICO / ICT region)

```
1. Coach opens "Schedule & Observe" tab
   → ScheduleDialog opens
   → Region auto-detected from profile → FICO framework selected
   → Fills in teacher, school, subject, grade, date → Save
   → cot_observations row created: status='Scheduled', framework='FICO'

2. Coach starts the observation
   → Status changes to 'Draft' (on first open of draft card)

3. Coach opens draft card → DC Analysis tab
   → Taps "Start Recording"
   → Browser microphone stream opened → MediaRecorder starts

4. Coach taps "Stop & Analyze"
   → MediaRecorder stops → audio Blob assembled
   → Calls dc-presign edge function → gets presigned_put_url + presigned_get_url + s3_key
   → Browser PUT audio blob directly to S3
   → Calls dc-start edge function
   → dc-start POSTs FormData to DC API → receives task_id
   → dc-start inserts dc_analyses row; updates cot_observations: dc_status='processing'
   → UI shows "DC is analysing the session…" with progress bar

5. DC analyses audio (async, typically 2–10 minutes)
   → DC POSTs progress updates to dc-callback
   → dc-callback updates dc_analyses.progress + current_step
   → Realtime fires → frontend progress bar updates

6. DC completes analysis
   → DC POSTs final results to dc-callback
   → dc-callback maps DC scores → fico_rubric
   → dc-callback calculates total_score + proficiency_level
   → dc-callback builds hots_notes from section feedback
   → All written to cot_observations in one UPDATE

7. Frontend receives completed event (Realtime or polling)
   → Refreshes cot_observations row
   → DCAnalysis tab shows collapsible results
   → FICO Rubric tab shows pre-filled Yes/Partial/No scores + DC banner
   → Debrief tab shows pre-filled observation notes + "Generated by DC" badge

8. Coach reviews, optionally edits any score or note, then submits
   → Submit button checks isFicoSectionBComplete → passes (DC filled these)
   → cot_observations.status = 'Submitted'
```

### HOTS flow (Punjab / Pindi)

DC Analysis tab shows "ICT Only" message — recording is unavailable. Coach scores manually via HotsRubricForm (sliders) and writes debrief notes manually. Submit requires `total_score > 0`.
